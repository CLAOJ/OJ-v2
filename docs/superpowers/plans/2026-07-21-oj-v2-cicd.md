# OJ-v2 CI/CD Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Test every change on push, publish container images to GHCR only on merge to `main`, and have the beta VPS pull-deploy them automatically via a label-scoped Watchtower.

**Architecture:** GitHub Actions (in the `OJ-v2` repo) runs unit tests + image builds; on `main` it pushes `ghcr.io/claoj/claoj-go` and `ghcr.io/claoj/claoj-web` (`:latest` + `:sha-<short>`). On the VPS, Watchtower (added via the `claoj-docker` repo's compose overlay) watches only the two v2 app containers and recreates them when the image digest changes. `v2_nginx` is made resilient to those recreates with a DNS resolver.

**Tech Stack:** GitHub Actions, Docker Buildx, GitHub Container Registry (ghcr.io), Watchtower (`containrrr/watchtower`), Docker Compose, nginx.

**Source of truth:** Design spec at `docs/superpowers/specs/2026-07-21-oj-v2-cicd-design.md`.

## Global Constraints

- **Two repos:** `OJ-v2` (`github.com/CLAOJ/OJ-v2`) holds source + the workflow; `claoj-docker` (`github.com/bachtam2001/claoj-docker`) holds the compose/deploy config the VPS runs.
- **Branch flow (OJ-v2):** all development on `dev`, pushed to `origin/dev`; merge to `main` only when stable. **Images publish only on `main`** (`push: ${{ github.ref == 'refs/heads/main' }}`).
- **Branch flow (claoj-docker):** work on branch `feat/v2-cicd`, then merge to `master` (claoj-docker's default branch — the one the VPS pulls). Use targeted `git add <files>` on every commit: the repo has a pre-existing dirty submodule (`claoj/repo`) that must never be swept into a commit.
- **Image names:** `ghcr.io/claoj/claoj-go`, `ghcr.io/claoj/claoj-web`. Private packages.
- **Tags:** `:latest` (moving, Watchtower-watched, main only) and `:sha-<short>` (immutable).
- **Build platform:** `linux/amd64` only.
- **Versions:** Go `1.25`, Node `20`.
- **CI test scope:** backend `go vet` + `go test` excluding `claoj/integration/` (DB-backed); web `npm run lint` + `npm test`. No service containers.
- **Watchtower isolation:** `WATCHTOWER_LABEL_ENABLE=true` + `WATCHTOWER_SCOPE=claoj-v2`; only `claoj_v2_backend` and `claoj_v2_web` carry the enable+scope labels. v1 stack and both judges are never touched.
- **Never touch:** env files (`local/v2.*.env`, `.env`) on the box; no DB DDL (v2 issues none; `v2_runtime_tables.sql` stays manual).
- **VPS user is root** (paths under `/root`); Watchtower reads `/root/.docker/config.json`.

## Task 1: CI/CD workflow (OJ-v2, branch `dev`)

**Files:**
- Create: `.github/workflows/ci-cd.yml`

**Interfaces:**
- Produces: images `ghcr.io/claoj/claoj-go` and `ghcr.io/claoj/claoj-web` tagged `:latest` (on `main`) and `:sha-<short>`. Consumed by the compose files in Tasks 3–4 and the cutover in Task 7.
- Consumes: nothing (uses the app Dockerfiles already in `claoj/` and `claoj-web/`, and the built-in `GITHUB_TOKEN`).

- [ ] **Step 1: Create the workflow file**

Create `.github/workflows/ci-cd.yml` with exactly this content:

```yaml
name: ci-cd

on:
  push:
    branches: [main, dev]
  pull_request:
    branches: [main, dev]
  workflow_dispatch:

concurrency:
  group: ci-cd-${{ github.ref }}
  cancel-in-progress: true

permissions:
  contents: read
  packages: write

env:
  IMAGE_BACKEND: ghcr.io/claoj/claoj-go
  IMAGE_WEB: ghcr.io/claoj/claoj-web

jobs:
  changes:
    runs-on: ubuntu-latest
    outputs:
      backend: ${{ steps.filter.outputs.backend }}
      web: ${{ steps.filter.outputs.web }}
    steps:
      - uses: actions/checkout@v4
      - uses: dorny/paths-filter@v3
        id: filter
        with:
          filters: |
            backend:
              - 'claoj/**'
              - '.github/workflows/ci-cd.yml'
            web:
              - 'claoj-web/**'
              - '.github/workflows/ci-cd.yml'

  test-backend:
    needs: changes
    if: needs.changes.outputs.backend == 'true'
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: claoj
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version: '1.25'
          cache-dependency-path: claoj/go.sum
      - name: go vet
        run: go vet ./...
      - name: go test (exclude DB-backed integration)
        run: go list ./... | grep -v '/integration' | xargs go test

  test-web:
    needs: changes
    if: needs.changes.outputs.web == 'true'
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: claoj-web
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: npm
          cache-dependency-path: claoj-web/package-lock.json
      - run: npm ci
      - run: npm run lint
      - run: npm test -- --ci

  build-push-backend:
    needs: [changes, test-backend]
    # success() is implicit for a status-function-free `if`, but this gate
    # publishes to prod on main — make "tests passed" explicit and unmissable.
    if: ${{ success() && needs.changes.outputs.backend == 'true' }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - name: Docker metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.IMAGE_BACKEND }}
          tags: |
            type=raw,value=latest,enable=${{ github.ref == 'refs/heads/main' }}
            type=sha,prefix=sha-,format=short
      - name: Build and push
        uses: docker/build-push-action@v6
        with:
          context: claoj
          platforms: linux/amd64
          push: ${{ github.ref == 'refs/heads/main' }}
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha,scope=backend
          cache-to: type=gha,mode=max,scope=backend

  build-push-web:
    needs: [changes, test-web]
    if: ${{ success() && needs.changes.outputs.web == 'true' }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - name: Docker metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.IMAGE_WEB }}
          tags: |
            type=raw,value=latest,enable=${{ github.ref == 'refs/heads/main' }}
            type=sha,prefix=sha-,format=short
      - name: Build and push
        uses: docker/build-push-action@v6
        with:
          context: claoj-web
          platforms: linux/amd64
          push: ${{ github.ref == 'refs/heads/main' }}
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha,scope=web
          cache-to: type=gha,mode=max,scope=web
```

## Task 2: nginx re-resolves upstreams (claoj-docker, branch `feat/v2-cicd`)

Prevents the 502 outage that would otherwise happen every time Watchtower recreates the backend/web container (new container IP, nginx caches the old one). See spec §7.1.

**Files:**
- Modify: `claoj/local/v2-nginx/conf.d/v2.conf`

**Interfaces:**
- Consumes: nothing. Produces: an nginx front that survives container recreates (relied on by Task 7's no-502 check).

- [ ] **Step 1: Create the `feat/v2-cicd` branch in claoj-docker**

```bash

git checkout -b feat/v2-cicd
```

(claoj-docker's default branch is `master`. Skip `git pull` if working offline; the local `master` is the base the spec was written against.)

- [ ] **Step 2: Replace `claoj/local/v2-nginx/conf.d/v2.conf` with the resolver version**

Full new file content:

```nginx
# Docker embedded DNS so nginx re-resolves upstreams after a container is
# recreated (e.g. by a Watchtower auto-deploy). Without this, nginx caches the
# upstream IP at worker start and 502s when the backend/web IP changes.
resolver 127.0.0.11 valid=10s ipv6=off;

server {
    listen 80 default_server;
    server_name _;
    client_max_body_size 256m;
    charset utf-8;

    # WebSocket (live submission updates) — must precede the /api/ prefix block
    location /api/events {
        set $v2_backend v2_backend;
        proxy_pass http://$v2_backend:8081$request_uri;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $http_host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 86400;
    }

    location /api/ {
        set $v2_backend v2_backend;
        proxy_pass http://$v2_backend:8081$request_uri;
        proxy_set_header Host $http_host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location / {
        set $v2_web v2_web;
        proxy_pass http://$v2_web:3000$request_uri;
        proxy_set_header Host $http_host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

> Note: using a variable in `proxy_pass` makes nginx defer DNS resolution to
> request time; appending `$request_uri` preserves the exact path + query
> (variable `proxy_pass` otherwise drops them). `server_name _` is unchanged.

```yaml
services:
  v2_watchtower:
    container_name: claoj_v2_watchtower
```

## Task 5: Rollback helper + README note (claoj-docker, `feat/v2-cicd`)

**Files:**
- Create: `claoj/scripts/rollback.sh`
- Modify: `README.md`

**Interfaces:**
- Consumes: tag vars `V2_BACKEND_TAG`/`V2_WEB_TAG` and image refs from Task 3.
- Produces: `scripts/rollback.sh <backend|web> <tag>` — pins a service to an immutable `:sha-` tag (freezing Watchtower) or back to `:latest` (resuming it).

- [ ] **Step 1: Create `claoj/scripts/rollback.sh`**

```bash
#!/usr/bin/env bash
# Pin a v2 service to a specific image tag, or return it to :latest.
#
#   scripts/rollback.sh backend sha-abc1234   # freeze backend at a known-good build
#   scripts/rollback.sh web     sha-abc1234
#   scripts/rollback.sh backend latest        # resume Watchtower auto-updates
#
# Persists the choice in .env (so later `docker compose up` keeps the pin) and
# recreates just that one service. Run on the VPS from the claoj-docker/claoj dir
# (or via its path — the script cd's to its own ../).
set -euo pipefail

cd "$(dirname "$0")/.."   # -> claoj-docker/claoj
ENV_FILE=".env"

SERVICE="${1:-}"; TAG="${2:-}"
if [[ -z "$SERVICE" || -z "$TAG" ]]; then
  echo "usage: rollback.sh <backend|web> <tag>" >&2; exit 2
fi

case "$SERVICE" in
  backend) SVC="v2_backend"; VAR="V2_BACKEND_TAG"; IMAGE="ghcr.io/claoj/claoj-go" ;;
  web)     SVC="v2_web";     VAR="V2_WEB_TAG";     IMAGE="ghcr.io/claoj/claoj-web" ;;
  *) echo "unknown service '$SERVICE' (expected 'backend' or 'web')" >&2; exit 2 ;;
esac

echo ">> pulling ${IMAGE}:${TAG}"
docker pull "${IMAGE}:${TAG}"

# upsert VAR=TAG in .env so the pin survives future compose ups
touch "$ENV_FILE"
if grep -q "^${VAR}=" "$ENV_FILE"; then
  sed -i "s|^${VAR}=.*|${VAR}=${TAG}|" "$ENV_FILE"
else
  echo "${VAR}=${TAG}" >> "$ENV_FILE"
fi

echo ">> recreating ${SVC} at ${TAG}"
docker compose -f docker-compose.yml -f docker-compose.v2.yml up -d --no-deps "$SVC"

if [[ "$TAG" == "latest" ]]; then
  echo ">> ${SVC}: Watchtower auto-updates RESUMED (:latest)"
else
  echo ">> ${SVC}: FROZEN at ${TAG} — run 'rollback.sh ${SERVICE} latest' to resume"
fi
```

```markdown
## 10. CI/CD (automated build + beta deploy)

**Pipeline (`.github/workflows/ci-cd.yml`).** Push/PR to `dev` or `main` runs, per
changed path: backend `go vet` + `go test` (excluding `./integration`), web
`npm run lint` + `npm test`, then a Docker image build of each changed app.
Images are **pushed to GHCR only on `main`** (`ghcr.io/claoj/claoj-go`,
`ghcr.io/claoj/claoj-web`), tagged `:latest` (moving) and `:sha-<short>`
(immutable). CI authenticates to GHCR with the built-in `GITHUB_TOKEN` — no secret
to manage.

**Branch flow.** Develop on `dev` (CI validates, publishes nothing); merge to
`main` to publish and deploy.

**Auto-deploy.** On the VPS, `containrrr/watchtower` (in
`claoj-docker/claoj/docker-compose.watchtower.yml`, scope `claoj-v2`) polls GHCR
every ~120 s and recreates only `claoj_v2_backend` / `claoj_v2_web` when a new
`:latest` digest appears. The v1 stack and both judges are never touched.

**One-time VPS setup.** Give the box read access to the private packages:

    echo <READ_ONLY_PAT> | docker login ghcr.io -u <github-username> --password-stdin

Watchtower mounts read-only.

**nginx.** `v2_nginx` uses a Docker DNS `resolver` + variable upstreams so it keeps
serving through a backend/web recreate (no 502, no manual restart).

**Rollback.** `claoj-docker/claoj/scripts/rollback.sh backend sha-<short>` pins a
service to an immutable build (freezing Watchtower); `... backend latest` resumes
auto-updates.
```

- Consumes: the workflow (Task 1), the compose/nginx/watchtower/rollback changes (Tasks 2–5), the docs (Task 6).
- Produces: a live auto-deploying beta.

- [ ] **Step 1: One-time GHCR login on the VPS**

Create a read-only PAT (`read:packages`) for a GitHub account with access to the `CLAOJ` org packages, then on the VPS:

```bash
echo <READ_ONLY_PAT> | docker login ghcr.io -u <github-username> --password-stdin
test -f /root/.docker/config.json && echo "auth file present"
```

Expected: `Login Succeeded` and `auth file present`.

- [ ] **Step 2: Publish images by merging OJ-v2 `dev` → `main`**

```bash
# in OJ-v2
git checkout main && git pull
git merge --no-ff dev -m "release: v2 ci/cd + auto-deploy"
git push origin main
```

Watch the `main` run in Actions. Expected: green, and under the repo's **Packages** the two images now show a `latest` tag and a `sha-<short>` tag. If the packages are created private, ensure the PAT user in Step 1 can read them (org package visibility / access).

- [ ] **Step 3: Merge the claoj-docker deploy changes to `master`**

```bash
# in claoj-docker (default branch is master)
git checkout feat/v2-cicd && git push origin feat/v2-cicd   # optional: open a PR
git checkout master && git pull
git merge --no-ff feat/v2-cicd -m "deploy: v2 ci/cd auto-deploy + nginx resolver"
git push origin master
```

- [ ] **Step 4: Pull the deploy repo on the VPS**

```bash
# on the VPS, in the claoj-docker checkout
git checkout master && git pull
cd claoj
```

- [ ] **Step 5: Bring up the new config (repoint images, reload nginx, start Watchtower)**

```bash
# pull the freshly published GHCR images
docker compose -f docker-compose.yml -f docker-compose.v2.yml pull v2_backend v2_web

# recreate v2 app services from the GHCR images + start Watchtower
docker compose -f docker-compose.yml -f docker-compose.v2.yml -f docker-compose.watchtower.yml \
  up -d v2_backend v2_web v2_watchtower

# apply the new nginx conf (bind-mounted; reload in place)
docker exec claoj_v2_nginx nginx -t && docker exec claoj_v2_nginx nginx -s reload
```

Expected: `v2_backend`/`v2_web` recreated from `ghcr.io/claoj/...:latest`,
`claoj_v2_watchtower` running, nginx reload `signal process started`.

- [ ] **Step 6: Verify auto-deploy end-to-end**

```bash
docker logs --tail 20 claoj_v2_watchtower      # shows it scanning; "Found new ... image" on updates
docker ps --format '{{.Names}}\t{{.Status}}'   # v1 containers keep their old uptime (untouched)
curl -s -o /dev/null -w "web %{http_code}\n"  https://beta.claoj.edu.vn/
curl -s -o /dev/null -w "api %{http_code}\n"  https://beta.claoj.edu.vn/api/problems
```

Expected: both `curl`s return `200` (the §7.1 no-502 check), only `claoj_v2_*`
were recreated, Watchtower is polling.

- [ ] **Step 7: Live smoke test**

Run the `docs/deployment.md` §6 smoke test against `https://beta.claoj.edu.vn`
(public reads, an auth round-trip, and a test submission reaching `D`/`AC`).

- [ ] **Step 8: Prove the loop**

Make a trivial visible change on OJ-v2 `dev`, merge to `main`, and confirm that
within ~2–3 min Watchtower recreates the relevant container and the change is live
on beta — with no manual step on the VPS.

- [ ] **Step 9: Rollback drill (optional but recommended once)**

```bash
# on the VPS, in claoj-docker/claoj
scripts/rollback.sh backend sha-<previous-short>   # pin to prior build
curl -s -o /dev/null -w "%{http_code}\n" https://beta.claoj.edu.vn/api/problems  # expect 200
scripts/rollback.sh backend latest                 # resume auto-updates
```

## Notes / deliberate refinements vs. the spec

- **Watchtower is a separate overlay** (`docker-compose.watchtower.yml`), not inside
  `docker-compose.v2.yml` as spec §7 sketched. Reason: keeping it in `v2.yml` would
  run Watchtower during local development, where it would overwrite locally-built
  images with the GHCR ones and fail to mount `/root/.docker/config.json`. The
  labels stay in `v2.yml` (harmless locally); only the service moved. Prod bring-up
  therefore layers three files.
- **Image tags are variabilised** (`:${V2_BACKEND_TAG:-latest}`) rather than a bare
  `:latest`, to make `rollback.sh` a clean one-liner. Default is still `latest`.
